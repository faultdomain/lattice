# GPU Observability & Cost Attribution

## Problem

Lattice provisions GPU resources (dedicated and fractional via Volcano vGPU) but has no visibility into GPU utilization after allocation. Operators cannot answer basic questions: Which GPUs are idle? What's our utilization rate? How much GPU-time did namespace X consume this month?

The GPU Operator already deploys DCGM Exporter (`dcgmExporter.enabled=true` in build.rs), so raw metrics are available on each node вҖ” but nothing collects, aggregates, or surfaces them through Lattice's monitoring stack.

## Goals

- Collect per-pod GPU utilization metrics into VictoriaMetrics (zero user config)
- Expose namespace-level and service-level GPU-hours for cost attribution
- Detect and alert on idle GPU allocations
- No new infrastructure вҖ” build on existing VictoriaMetrics + KEDA stack

## Non-Goals

- Dollar-cost calculation (requires cloud pricing APIs, out of scope)
- GPU scheduling optimization (Volcano handles this)
- AMD/Intel GPU support (NVIDIA only, matching existing stack)

## Detailed Design

### Phase 1: DCGM Metric Collection

DCGM Exporter is already deployed by the GPU Operator but not scraped by vmagent. We need a VMPodScrape (not VMServiceScrape вҖ” DCGM runs as a DaemonSet, not a Service).

**New resource generated by `lattice-infra/src/bootstrap/gpu.rs`:**

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMPodScrape
metadata:
  name: dcgm-exporter
  namespace: gpu-operator
spec:
  podMetricsEndpoints:
    - port: gpu-metrics      # DCGM exporter port name
      path: /metrics
      interval: 15s          # GPU metrics change fast
  selector:
    matchLabels:
      app: nvidia-dcgm-exporter
  namespaceSelector:
    matchNames:
      - gpu-operator
```

**Key DCGM metrics collected:**

| Metric | Description | Use |
|--------|-------------|-----|
| `DCGM_FI_DEV_GPU_UTIL` | GPU compute utilization % | Utilization tracking |
| `DCGM_FI_DEV_FB_USED` | Framebuffer memory used (MiB) | Memory tracking |
| `DCGM_FI_DEV_FB_FREE` | Framebuffer memory free (MiB) | Capacity planning |
| `DCGM_FI_DEV_GPU_TEMP` | GPU temperature (C) | Health monitoring |
| `DCGM_FI_DEV_POWER_USAGE` | Power draw (W) | Power/cost tracking |
| `DCGM_FI_DEV_SM_CLOCK` | SM clock frequency (MHz) | Throttling detection |
| `DCGM_FI_DEV_MEM_CLOCK` | Memory clock frequency (MHz) | Throttling detection |

These metrics include labels: `gpu` (device index), `UUID`, `device`, `modelName`, `Hostname`, plus standard k8s pod labels injected by vmagent relabeling.

### Phase 2: Pod-Level GPU Attribution

DCGM metrics are per-device, not per-pod. For multi-tenant attribution we need to join GPU device metrics with pod ownership. The DCGM Exporter already provides `DCGM_FI_DEV_*` metrics with `pod` and `namespace` labels when running with the `--kubernetes` flag (enabled by default in the GPU Operator).

**Verification**: The GPU Operator Helm chart sets `dcgmExporter.env[DCGM_EXPORTER_KUBERNETES]=true`, which enables pod-label enrichment. No changes needed.

If pod labels are missing (older GPU Operator versions), we fall back to a recording rule that joins on node + GPU index:

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMRule
metadata:
  name: gpu-pod-attribution
  namespace: monitoring
spec:
  groups:
    - name: gpu.pod_attribution
      interval: 30s
      rules:
        # GPU utilization per pod (fallback join if pod labels missing)
        - record: lattice:gpu:pod_utilization
          expr: |
            DCGM_FI_DEV_GPU_UTIL
            * on(namespace, pod) group_left(container)
            kube_pod_container_resource_requests{resource="nvidia.com/gpu"}
```

### Phase 3: Namespace & Service Aggregation

Recording rules that roll up per-pod GPU metrics into namespace-level and service-level aggregates, suitable for cost attribution dashboards.

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMRule
metadata:
  name: gpu-cost-attribution
  namespace: monitoring
spec:
  groups:
    - name: gpu.cost_attribution
      interval: 1m
      rules:
        # Average GPU utilization per namespace (gauge)
        - record: lattice:gpu:namespace_avg_utilization
          expr: |
            avg by (namespace) (DCGM_FI_DEV_GPU_UTIL{pod!=""})

        # GPU-seconds consumed per namespace (counter, for gpu-hours calculation)
        # Each sample = (number of GPUs allocated) * (scrape interval seconds)
        - record: lattice:gpu:namespace_gpu_seconds_total
          expr: |
            sum by (namespace) (
              count by (namespace, pod) (DCGM_FI_DEV_GPU_UTIL{pod!=""})
            ) * 60

        # GPU-seconds per service (using app.kubernetes.io/name label)
        - record: lattice:gpu:service_gpu_seconds_total
          expr: |
            sum by (namespace, label_app_kubernetes_io_name) (
              count by (namespace, pod, label_app_kubernetes_io_name) (
                DCGM_FI_DEV_GPU_UTIL{pod!=""}
              )
            ) * 60

        # GPU memory utilization per namespace (%)
        - record: lattice:gpu:namespace_avg_memory_utilization
          expr: |
            avg by (namespace) (
              DCGM_FI_DEV_FB_USED / (DCGM_FI_DEV_FB_USED + DCGM_FI_DEV_FB_FREE) * 100
            )
```

**GPU-hours query example** (for dashboards or API):

```promql
# GPU-hours consumed by namespace "ml-training" in the last 24h
increase(lattice:gpu:namespace_gpu_seconds_total{namespace="ml-training"}[24h]) / 3600
```

### Phase 4: Idle GPU Detection & Alerting

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMRule
metadata:
  name: gpu-idle-alerts
  namespace: monitoring
spec:
  groups:
    - name: gpu.idle_detection
      interval: 1m
      rules:
        # GPU allocated but utilization < 5% for 15 minutes
        - alert: GpuIdlePod
          expr: |
            DCGM_FI_DEV_GPU_UTIL{pod!=""} < 5
            and on(namespace, pod)
            kube_pod_container_resource_requests{resource="nvidia.com/gpu"} > 0
          for: 15m
          labels:
            severity: warning
          annotations:
            summary: "GPU idle on {{ $labels.pod }} in {{ $labels.namespace }}"
            description: "GPU {{ $labels.gpu }} allocated to pod {{ $labels.pod }} has been below 5% utilization for 15 minutes."

        # Namespace-level: average utilization below 10% for 30 minutes
        - alert: GpuNamespaceUnderutilized
          expr: lattice:gpu:namespace_avg_utilization < 10
          for: 30m
          labels:
            severity: info
          annotations:
            summary: "Low GPU utilization in namespace {{ $labels.namespace }}"
            description: "Average GPU utilization in {{ $labels.namespace }} has been below 10% for 30 minutes."

        # Thermal throttling detection
        - alert: GpuThermalThrottling
          expr: DCGM_FI_DEV_GPU_TEMP > 85
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "GPU thermal throttling on {{ $labels.Hostname }}"
            description: "GPU {{ $labels.gpu }} temperature is {{ $value }}C, likely throttling."
```

### Phase 5: GPU-Aware KEDA Autoscaling

Expose GPU utilization as a KEDA scaling metric so LatticeModel and LatticeService autoscaling can react to GPU load, not just CPU/memory.

Users already declare custom metrics in `AutoscalingSpec.metrics`. With DCGM metrics flowing into VictoriaMetrics, users can reference them directly:

```yaml
# In a LatticeModel or LatticeService spec
autoscaling:
  max: 8
  metrics:
    - metric: DCGM_FI_DEV_GPU_UTIL
      target: 80
```

The service compiler already handles custom Prometheus metrics via KEDA's `prometheus` trigger. No compiler changes needed вҖ” the metric name is passed through to a KEDA ScaledObject targeting vmselect.

## Implementation Plan

### Step 1: VMPodScrape for DCGM (lattice-infra)

- Add `generate_gpu_monitoring()` function to `bootstrap/gpu.rs`
- Generate VMPodScrape targeting `nvidia-dcgm-exporter` pods
- Conditional on `LatticeClusterSpec.gpu && monitoring.enabled`
- Call from `generate_core()` alongside existing GPU stack

### Step 2: Recording Rules (lattice-infra)

- Add `generate_gpu_recording_rules()` to `bootstrap/prometheus.rs`
- Generate VMRule CRDs for pod attribution, namespace rollup, service rollup
- Conditional on GPU + monitoring enabled

### Step 3: Alert Rules (lattice-infra)

- Add `generate_gpu_alerts()` to `bootstrap/prometheus.rs`
- Generate VMRule CRDs for idle detection, thermal throttling
- Same conditional as recording rules

### Step 4: LMM for DCGM Exporter (lattice-infra)

- Add LatticeMeshMember for `nvidia-dcgm-exporter` in `gpu-operator` namespace
- Allow vmagent to scrape metrics port through mesh
- Required for mTLS environments

### Step 5: Documentation & Testing

- Integration test: deploy GPU workload, verify metrics appear in vmselect
- Verify idle alert fires for a pod that allocates but doesn't use GPU
- Verify GPU-hours recording rule produces correct values

## CRD Changes

None. This feature is purely infrastructure вҖ” no new CRDs or spec changes. GPU metrics flow through the existing VictoriaMetrics pipeline and are queryable via the existing vmselect API.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| DCGM Exporter missing pod labels | Fallback recording rule joins on node + device index |
| High cardinality from per-GPU metrics | 15s scrape interval, 24h retention (existing VM config) |
| vGPU fractional metrics inaccuracy | DCGM reports physical GPU util; document that fractional shares see the same device metric |
| Alert fatigue from idle detection | Conservative thresholds (5% for 15min), `info` severity for namespace-level |
